QubicKit is the install-first entrypoint for building on the Qubic network. The monorepo publishes four packages (`core`, `sdk`, `cli`, `web`) that share transports, schemas, and tooling so you can adopt them à la carte without cloning this repo. These docs walk through everything you need to ship production integrations, from the lowest-level HTTP clients up through wallet orchestration, CLIs, and web dashboards.

## Install QubicKit (no clone required)

```bash
# pick the package that matches your use case
bun add @qubickit/core         # transports, transactions, identity helpers
bun add @qubickit/sdk          # wallet/session orchestration (when ready)
bun add @qubickit/cli          # CLI as a dependency (or `bun add -g @qubickit/cli`)
bun add @qubickit/web          # React hooks/components (when ready)
```

- **Core** is production-ready today—drop it into scripts, services, or CLIs.
- **SDK/CLI/Web** will reach parity in upcoming phases; install them when you need their capabilities.

Only contributors need to clone the monorepo; everyone else installs what they need from npm and references these docs.

## How these docs are organized

- **Getting started** explains the npm install flow first, then covers optional monorepo scripts if you plan to contribute.
- **Core** already contains end-to-end docs (overview, client reference, transactions, utilities) for `@qubickit/core`.
- **SDK, CLI, and Web** folders currently outline what we are building next so we can add full guides without reshuffling the IA later.

## Delivery phases

The roadmap from `planning/docs.md` drives how and when new sections unlock:

- **Phase 0 – Platform setup**: Fumadocs on React Router, Bun scripts (`bun dev`, `bun run build`, lint/test), shared MDX conventions, and CI hooks for spell checks + broken links.
- **Phase 1 – MVP content**: Fundamentals, quickstarts for every package, registry reference material, and FAQ/troubleshooting.
- **Phase 2 – Reference automation**: Generated API docs (TSDoc + Zod metadata), deep dives (transaction lifecycle, contract encoding, watcher architecture), search, versioning, and diagram libraries.
- **Phase 3 – Stable hub**: Localization scaffolding, analytics, changelog automation, cookbook recipes, governance, and release playbooks.
